这次来看看CRI (Container Runtime Interface)和CNI (Container Networking Interface)的关系。

CRI

Background

Kubernetes作为容器编排引擎，它在Worker Nodes上会启用Kubelet，来管理节点上各个容器的create、list、watch和delete。而具体到节点上真正创建的容器形式，K8s希望以开放的形式支持更多类型的container runtime，比如Docker默认采用的runC，或者之前笔者使用的Kata Containers。

这些container runtime，可以基于Linux namespace、cgroups创建的轻量虚拟化隔离环境，也可以是像Kata这样创建的VM；当然这种自由度最终也是受限的，想要在K8s上使用、被Kubelet调用，就需要支持K8s CRI。

最初Kubernetes内置了Docker的调用接口，而后在1.3版本又集成了rkt作为runtime的接口。为了解耦container runtime与Kubelet的代码逻辑，减轻双方代码变更的维护负担，K8s于1.5版本引入了CRI，从此Kubelet就可以直接调用接口定义的容器操作，而不必关心runtime的具体实现。

Architecture

如上图所示，CRI为Kubelet和container runtime直接的通信设计了一对gRPC接口，Kubelet通过gRPC与CRI shim通信，CRI shim进而控制runtime操纵容器。

CRI shim是一个接口转换层，我们熟知的docker-shim、containerd-shim即为它的实现。
在Docker的方案中，dockershim是独立的进程，而在有些runtime比如PouchContainer中则是内嵌的代码模块。
以K8s 1.14的代码为例，Kubelet启动时，在NewMainKubelet()函数内会判断默认使用的CRI runtime，如果是Docker，则为dockershim创建CRI shim的gRPC server（dockershim.NewDockerService()）并启动。

Kubelet内置了CRI的gRPC client，Kubelet对CRI runtime的调用，就被转换成对CRI shim的gRPC请求。

K8s CRI的接口分为ImageService和RuntimeService两类。
前者负责管理容器镜像，后者负责容器/Pod的生命周期管理、及与容器的交互。
下面我们主要关注RuntimeService，它在CRI与CNI的交互间起到主要作用。

CNI

CNI是由Google和CoreOS主导的容器组网规范，被Kubernetes、Mesos、Cloud Foundry等项目采用，目前也是CNCF的重要项目之一。

Overview

不管是Docker还是Kubernetes，都尚不存在完美、普适性的网络方案。CNI的理念与CRI相似，它旨在为容器平台提供网络的标准化接口，允许不同容器平台通过相同的接口调用不同的网络组件；网络组件包括flannel、calico、openvswitch、ipvlan等大家耳熟能详的方案，也包括类似CNM中的bridge等方案。

简而言之，CNI是一个接口协议，连接了容器管理系统（比如K8s）和网络插件（比如flannel、calico）。

Kubernetes和CNI Plugins之间，会以JSON格式的文件进行通信，完成容器组网的ADD、DEL/DELETE和CHECK等行为，为容器设置网络namespace（netNS）、加入或卸载网卡。

类似CNI，Docker以libnetwork的形式提供了另一套名为CNM (Container Networking Model)的组网规范，关于这二者在workflow上的区别以及具体规范可以参考容器网络规范 CNM vs. CNI和参考资料2。

CNI Plugins

CNI官方提供了三种类型的CNI Plugins，即main、meta和ipam，对应负责容器组网的不同层面工作。

  • main
    • loopback：为容器生成lo网卡；
    • bridge：类似Docker的bridge模式，将所有容器连到vswitch上；
    • macvlan：使用macvlan技术，从单个物理网卡中虚拟出多个拥有独立IP、MAC地址的虚拟网卡（并非SR-IOV）；
    • ipvlan：和macvlan类似，但虚拟网卡拥有相同的MAC地址。
  • meta
    • flannel：flannel插件分配网段，再调用bridge插件，以创建overlay网络。
  • ipam
    • host-local：基于本地文件，进行IP分配、管理以及缓存；
    • dhcp：从已运行的DHCP服务器中获取IP地址。
